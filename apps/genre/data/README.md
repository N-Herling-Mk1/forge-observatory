# `data/` — GTZAN data area & genealogy

This folder is the **genealogy source of truth** for the genre project. The website's
"project genealogy" panel is built from `manifest.json` emitted here — so the data area
isn't just storage, it's documentation that stays in sync with reality.

## Layout
```
data/
├── README.md          ← you are here
├── raw/               # GTZAN, patched in place — GITIGNORED (1.2 GB, never commit)
├── before/            # pre-fix EDA snapshot   — COMMITTED (eda_stats.json)
├── after/             # post-fix EDA snapshot  — COMMITTED (eda_stats.json + repair note)
├── interim/           # cached mel-spectrograms / 58-feat tables — GITIGNORED
├── manifest.json      # EMITTED genealogy: source, license, checksums, counts, splits
└── manifest.fixed.json# EMITTED repair receipt (what data_doctor.py changed + hashes)
```

`raw/` holds the actual dataset bytes and is what training + EDA read from
(`run_eda.py` defaults `--data-root` to `data/raw`). `before/` and `after/` are small
EDA *snapshots* (the `eda_stats.json` the dashboard renders), not data — `before` =
stats on the data prior to repair, `after` = stats after it.

## Fetching GTZAN
```bash
# from repo root
mkdir -p projects/genre/data/raw
# GTZAN (Tzanetakis 2002). Mirror URLs rot; the Kaggle "GTZAN Dataset - Music Genre
# Classification" mirror is the most reliable and ships the 58-feature CSVs + mel images.
# Place audio under data/raw/genres_original/<genre>/*.wav (also images_*/ and the CSVs).
```
> Do **not** commit `raw/` or `interim/`. They're gitignored. Genealogy travels via
> `manifest.json` (small, committed), not the bytes.

## Repairs applied (jazz.00054)
The EDA surfaced one genuinely broken track. Rather than build a parallel `fixed/`
tree, the two repaired files were **patched directly into `raw/`** so the data loader
has a single read path:

| file | repair |
| ---- | ------ |
| `raw/genres_original/jazz/jazz.00054.wav`  | corrupt original replaced with a known-good 27.38 s clip |
| `raw/images_grey_scale/jazz/jazz00054.png` | missing grey regenerated (128×128 L) by `src/data_doctor.py`, histogram-matched to the existing jazz greys (mean ~116; corpus 120 ± 5) so it isn't a brightness outlier |

`data_doctor.py` writes `manifest.fixed.json` recording both actions + sha256 hashes,
and `after/README.txt` carries the note next to the snapshot.

**Consequence to be honest about:** `raw/` is no longer byte-for-byte the original
GTZAN — it was patched in place. The pre-fix state is preserved only in
`before/eda_stats.json`; the `after` snapshot reflects the patched data.

**Not altered — documented, not fabricated** (these are real properties of the data,
not defects to impute):
- 10 tracks have only 9 × 3-sec segments (clip slightly under 30 s). Kept as-is; the
  3-sec rows are independent training samples, so a short track simply contributes 9.
- 10 hiphop clips run 30.649 s instead of 30.0 s. Left untrimmed (the 3-sec path uses
  only the first 30 s regardless).

## What `manifest.json` captures (genealogy)
- `source`: URL + Tzanetakis & Cook 2002 citation
- `license`: GTZAN usage terms
- `n_files`, `genres`, per-class counts (expect 100 each)
- `sample_rate` (22050), `channels` (mono), `bit_depth`
- `known_issues`: e.g. `["jazz.00054 corrupt (repaired — see manifest.fixed.json)",
  "Sturm 2013: duplicates/mislabels/artist-repeat"]`
- `split`: `{ mode, seed, train/val/test sizes }` — recorded so a run is reproducible
- `checksums`: per-file or per-archive hashes → detects silent data drift

`manifest.json` is regenerated by the data adapter; the EDA step reads it. Treat it as
generated, not hand-edited. `manifest.fixed.json` is likewise emitted by `data_doctor.py`.

## Why genealogy matters here specifically
Reproduction numbers are only meaningful relative to *which* GTZAN and *which* split.
Two papers reporting "GTZAN accuracy" may not be comparable. The manifest pins down
exactly what we ran on — including the jazz.00054 patch — so our scorecard
("ours vs paper") is honest.
